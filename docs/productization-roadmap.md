## 2. 距离真实上线还缺什么

按重要性排序：

### 2.1 必须补齐（阻塞上线）

| # | 缺失项 | 影响 | 优先级 |
|---|--------|------|--------|
| 1 | **HTTPS + 域名** | JWT token 和密码明文传输，不可接受 | P0 |
| 2 | **数据库自动备份 + 恢复演练** | 数据丢失无补救 | P0 |
| 3 | **生产环境监控与告警** | 服务挂了不知道 | P0 |
| 4 | **员工账号数量限制** | 当前不限制，无法按套餐收费 | P0 |
| 5 | **工单数量/存储限制** | 无法按用量控制成本 | P1 |
| 6 | **日志轮转与清理** | Docker 日志撑爆磁盘 | P1 |
| 7 | **首次部署后强制改密** | 默认密码 `Admin@2024` 仍是安全风险 | P1 |
| 8 | **admin 账号名可预测** | 所有租户的 ADMIN 都叫 `admin`，暴力枚举风险 | P1 |

### 2.2 应该补齐（影响体验和运维）

| # | 缺失项 | 说明 |
|---|--------|------|
| 9 | 邮件/短信通知 | 派单、状态变更无法主动触达客户和师傅 |
| 10 | 数据导出（Excel） | 客户要求导出工单报表时无法满足 |
| 11 | 图片上传（报修/完工） | 当前仅支持 txt/md 文档，不支持图片附件 |
| 12 | 微信订阅消息 | 小程序无法推送派单通知 |
| 13 | 租户自助修改企业信息 | 当前企业名称等需平台管理员修改 |
| 14 | API 文档（Swagger/Knife4j） | 对外提供 API 时无文档 |
| 15 | 健康检查完善（含 DB/Redis/Qdrant 连通性） | 当前仅返回 UP，不检查依赖 |

### 2.3 商业化补齐（影响收费）

| # | 缺失项 | 说明 |
|---|--------|------|
| 16 | 套餐定义与自动执行 | 当前限额字段靠人工配置，无套餐模板 |
| 17 | 试用到期自动通知 | 到期前无提醒，直接拒绝访问体验差 |
| 18 | 用量看板（租户侧） | 租户看不到自己用了多少额度 |
| 19 | 平台运营看板 | 平台方看不到各租户用量和健康度 |
| 20 | 支付集成 | 无法在线支付，需人工收款 |

## 3. V0.7.0 — Beta 上线准备

**目标：** 可以在一台公网服务器上安全运行，支撑平台方内部使用和给 1-2 个朋友团队试用。

**时间预估：** 2-3 周

### 3.1 安全加固

- [ ] **HTTPS 强制**：Nginx 配置 SSL 终结 + HSTS，HTTP 一律 301 到 HTTPS
- [ ] **首次部署强制改密**：`superadmin` 首次登录检测默认密码，不修改则无法进入其他页面
- [ ] **admin 账号名随机化**：创建租户时不固定为 `admin`，改为可配置或随机后缀
- [ ] **JWT secret 部署时强制替换**：check-env.sh 检测默认值则拒绝继续
- [ ] **MySQL/Redis/Qdrant 端口不暴露公网**：docker-compose 确认 `127.0.0.1` 绑定

### 3.2 运维基础

- [ ] **定时备份自动化**：crontab 模板 + 备份保留策略（7 天日备 + 4 周周备）
- [ ] **备份恢复演练脚本**：restore-mysql.sh 增加自动恢复验证步骤
- [ ] **日志轮转**：Docker daemon 配置 `max-size=50m max-file=5`，或容器内 logrotate
- [ ] **健康检查端点完善**：`/api/health` 返回 DB/Redis/Qdrant 连通状态
- [ ] **Uptime 监控**：至少接入一个外部监控（UptimeRobot 免费版或自建 Healthchecks）
- [ ] **异常重启告警**：Docker `restart: always` + 重启次数监控

### 3.3 功能完善

- [ ] **员工数量限制**：tenant 表新增 `max_users`，创建员工时校验
- [ ] **工单附件（图片）**：后端 `ticket_attachment` 表 + 上传接口，前端图片上传组件
- [ ] **数据导出**：工单列表导出 Excel（Apache POI），按筛选条件导出
- [ ] **平台运营看板**：租户数量、活跃租户、今日工单总量、AI 调用总量

### 3.4 部署简化

- [ ] **一键部署脚本**：`deploy/scripts/setup.sh` 覆盖：check-env → build → up → init-demo → 输出访问地址
- [ ] **.env 交互式配置向导**：引导填写必填项（域名、密码、API Key）
- [ ] **部署后自动输出访问信息**：管理后台 URL、平台管理员账号密码（提示立即修改）

### 3.5 文档

- [ ] **运维手册**：日常巡检、日志查看、备份恢复、证书续期
- [ ] **租户管理员使用手册**：后台功能说明、知识库配置、工单流程
- [ ] **故障排查 FAQ**：常见问题及解决步骤

### V0.7.0 完成标准

- [ ] HTTPS 全站加密
- [ ] 备份自动化 + 至少一次恢复演练成功
- [ ] 公网服务器连续运行 7 天无异常重启
- [ ] 健康检查 + 外部监控正常运行
- [ ] 员工数量限制生效
- [ ] 一键部署脚本可用（在新服务器上从零部署成功）

## 7. 租户生命周期设计

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  试用期   │ → │  活跃付费  │ → │  到期提醒  │ → │  宽限期   │ → │  已暂停   │
│  TRIAL   │   │  ACTIVE   │   │  EXPIRING  │   │  GRACE    │   │  SUSPENDED│
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
      ↓              ↓                                               ↓
      ↓              ↓                                          ┌──────────┐
      └──────────────┴────────────────────────────────────────→ │  已关闭   │
         (任何阶段可主动取消)                                      │  CLOSED   │
                                                                 └──────────┘
```

### 7.1 各阶段定义

| 阶段 | 说明 | 系统行为 |
|------|------|---------|
| **试用期 TRIAL** | 新注册，默认 7 天试用 | 所有功能正常，门户顶部显示"试用剩余 X 天" |
| **活跃付费 ACTIVE** | 已付费，在有效期内 | 全功能正常，无干扰提示 |
| **到期提醒 EXPIRING** | 距离到期 ≤7 天 | 门户+后台顶部黄色横幅提醒续费 |
| **宽限期 GRACE** | 到期后 3 天内 | 功能正常但门户顶部红色横幅 + 登录后弹窗提醒 |
| **已暂停 SUSPENDED** | 宽限期过后未续费 | 拒绝登录和公开接口，数据保留，管理后台显示"已暂停" |
| **已关闭 CLOSED** | 主动取消或暂停超过 30 天 | 数据保留 30 天后可清理，不可恢复 |

### 7.2 当前已实现 vs 待实现

| 阶段 | 当前状态 |
|------|---------|
| expired_at 全链路拦截 | ✅ V0.3.4 已实现 |
| 到期拒绝登录 | ✅ V0.3.4 |
| 到期拒绝公开接口 | ✅ V0.3.4 |
| 门户"服务已到期"提示 | ✅ V0.3.4 |
| 试用到期提醒（提前 N 天） | ❌ V0.8.0 |
| 宽限期 | ❌ V0.9.0 |
| 到期后数据保留策略 | ❌ V0.9.0 |
| 主动取消流程 | ❌ V0.9.0 |
| 租户状态字段（TRIAL/ACTIVE/SUSPENDED/CLOSED） | ⚠️ 当前仅 `status=ACTIVE/DISABLED`，需扩展 |

|-----------|---------|

### 8.3 额度超限处理原则

- 超限时返回明确错误码和提示："员工数量已达上限（10/10），如需扩容请联系客服或升级套餐"
- 不静默拒绝，不返回 500
- 平台方可为单个租户临时调高限额（不改变套餐）

## 11. 运维/备份/监控要求

### 11.1 备份策略

| 数据类型 | 工具 | 频率 | 保留 | 异地 |
|---------|------|------|------|------|
| MySQL | mysqldump + gzip | 每天 3:00 | 7 天日备 + 4 周周备 | rsync 到另一台机器或云存储 |
| 上传文件 | tar.gz | 每周日 4:00 | 4 周 | 同上 |
| Qdrant | tar.gz snapshot | 每周日 4:30 | 4 周 | 同上 |

**备份验证：** 每月至少一次恢复演练，确保备份可恢复。

### 11.2 监控指标

| 层级 | 指标 | 告警阈值 | 工具建议 |
|------|------|---------|---------|
| 服务存活 | `/api/health` 可达 | 连续 2 次失败 | UptimeRobot（免费） |
| 容器状态 | 全部 Running | 任一非 running | `docker ps` + cron 脚本 |
| 磁盘 | 使用率 | >80% 告警，>90% 紧急 | `df -h` + cron |
| 内存 | 使用率 | >90% 告警 | `free -h` + cron |
| API 错误率 | 5xx 占比 | >5% 告警 | Nginx access log + 脚本 |
| 慢查询 | 执行 >1s | 任一出现即通知 | MySQL slow query log |
| SSL 证书 | 到期天数 | <30 天告警 | `certbot certificates` + cron |
| 备份成功 | 备份文件存在 + 大小 > 0 | 备份失败立即告警 | cron 执行后检查 |

### 11.3 告警通知渠道

按优先级：
1. **紧急（P0）**：服务挂了、磁盘满了 → 电话/短信/企业微信
2. **重要（P1）**：备份失败、证书快到期 → 邮件 + 企业微信
3. **提醒（P2）**：用量接近上限 → 邮件

V0.7.0 先用最简单的方案：cron + curl 健康检查失败时发邮件。

### 11.4 日志管理

- Docker 容器日志：`max-size=50m, max-file=5`
- Java 应用日志：保留 30 天，按天滚动
- Nginx access log：保留 30 天，按天滚动
- 不收集客户端 IP 到应用日志（隐私考虑，可选）

## 12. 首批目标客户画像

### 12.1 理想首批客户特征

| 维度 | 描述 |
|------|------|
| 行业 | 家电维修（空调/热水器/净水器/厨电） |
| 规模 | 3-15 人的维修团队，有 1-2 个客服 + 3-10 个师傅 |
| 痛点 | 报修靠微信/电话，经常漏单；师傅去了哪里老板不知道 |
| 技术能力 | 会用微信、会扫码、会填表单，但不一定用过 SaaS |
| 决策人 | 老板或店长，关心减少漏单、提高效率 |
| 付费意愿 | 愿意为"减少漏单"每月付 99-299 元 |
| 获取渠道 | 熟人介绍、维修行业微信群、本地维修圈子 |

### 12.2 首批客户不该找的

- 大型连锁售后（需求复杂、定制多、决策链长）
- 完全没有电脑的团队（只有手机，管理后台用不了）
- 对价格极度敏感的个人维修师傅（不愿意付月费）
- 需要私有化部署的客户（只做 SaaS）

### 12.3 获取首批客户的方式

1. **熟人网络**：朋友/前同事的维修团队
2. **行业微信群**：维修师傅群、家电售后群
3. **本地维修店**：上门拜访或电话联系
4. **免费试用换反馈**：前 5 个客户免费使用 3 个月，条件是每周给反馈

### 12.4 首批客户数量目标

- V0.7.0：1-2 个内部朋友团队
- V0.8.0：3-5 个真实试用客户
- V1.0.0：5+ 个付费客户

## 13. 试用客户交付流程

### 13.1 交付前准备（平台方）

- [ ] 确认服务器稳定运行（至少 7 天无异常）
- [ ] 准备租户演示数据（空调/热水器常见 FAQ）
- [ ] 准备客户行业的知识库模板（可快速导入）
- [ ] 打印/准备租户管理员使用手册（一页纸）

### 13.2 交付当天（约 1 小时）

```
1. 创建租户（10 分钟）
   - 平台管理后台创建租户
   - 填写企业名称、联系电话
   - 选择"试用版"套餐
   - 设置 30 天试用期

2. 交付账号（5 分钟）
   - 发送管理后台地址 + 企业编码 + 管理员账号密码
   - 提醒客户首次登录后修改密码

3. 引导客户首次配置（30 分钟，远程或现场）
   - 登录管理后台
   - 修改密码
   - 设置企业门户（名称、电话、主题色）
   - 创建知识库 → 导入 FAQ 模板 → 测试 AI 问答
   - 创建员工账号（客服 + 师傅）
   - 测试完整流程：门户报修 → 后台派单 → 师傅处理 → 查询进度

4. 交付材料（5 分钟）
   - 管理后台地址 + 账号
   - 门户地址（可生成二维码打印出来贴店里）
   - 师傅端登录说明（手机浏览器 或 小程序）
   - 使用手册（电子版）
   - 技术支持联系方式

5. 确认客户可以自行操作（10 分钟）
   - 客户独立完成一次报修→派单→处理→查询全流程
```

### 13.3 试用期间跟进

| 时间 | 动作 | 目的 |
|------|------|------|
| 第 1 天 | 确认已登录并改密 | 安全 + 确认可用 |
| 第 3 天 | 确认已配置知识库、至少 1 个工单 | 确认在用 |
| 第 7 天 | 回访：遇到什么问题？有什么建议？ | 收集反馈 |
| 第 14 天 | 回访：数据积累如何？师傅用得习惯吗？ | 深度使用反馈 |
| 第 21 天 | 提前告知试用即将到期，询问续费意向 | 转化 |
| 第 30 天 | 到期处理：续费→升级套餐，不续→宽限期→暂停 | 决策 |

### 13.4 常见客户问题预案

| 问题 | 回答 |
|------|------|
| "AI 回答不太准" | 知识库内容需要补充，帮你们整理一下 FAQ 模板 |
| "师傅不想用" | 师傅端只需看工单+点按钮，比微信翻聊天记录简单 |
| "能不能加 XX 功能" | 记录到需求池，如果多个客户需要优先做 |
| "能不能便宜点" | 试用期免费，早期客户有优惠 |
| "数据安全吗" | 数据隔离、定时备份、HTTPS 加密 |

## 14. 暂不做事项

以下明确 **不在 V1.0 范围内**，避免产品失控：

### 14.1 功能层面

- ❌ 私有化部署（只做 SaaS 多租户）
- ❌ 多语言/国际化（只做中文）
- ❌ 自定义域名 per 租户（所有租户共用平台域名）
- ❌ 大屏驾驶舱 / BI 看板
- ❌ 复杂审批流（多级审批、会签）
- ❌ 师傅抢单模式
- ❌ 路线规划 / 地图集成
- ❌ 库存管理 / 配件进销存
- ❌ 财务结算系统（只记录费用备注，不做对账结算）
- ❌ 客户评价 / 满意度调查
- ❌ 视频教程平台
- ❌ 开放 API / Webhook（V1.0 仅提供 API 文档，不做开放平台）

### 14.2 技术层面

- ❌ 微服务拆分（保持模块化单体）
- ❌ Kubernetes / Helm（保持 Docker Compose）
- ❌ 多机房 / 异地多活
- ❌ 自建 CDN
- ❌ Elasticsearch / 日志中心（保持文件日志 + Docker logs）
- ❌ Prometheus + Grafana（V1.0 用更轻量的方案）
- ❌ CI/CD 自动部署（保持手动部署 + 脚本）

### 14.3 商业层面

- ❌ 免费套餐（只有试用版免费 7 天，无永久免费版）
- ❌ 广告投放获客
- ❌ 代理商/渠道体系
- ❌ 定制开发服务
- ❌ 7×24 小时 SLA

### 14.4 后续版本可考虑

以下功能如果多个付费客户需要，可在 V1.1+ 评估：

- 微信手机号授权（免填）
- 微信 openid 绑定师傅
- 客户评价与满意度
- 配件库存简单管理
- PDF/Word 文档上传解析
- 多网点/多门店支持
- 工单自动分配（按师傅位置/负载）
- API Webhook（工单状态变更通知第三方）
